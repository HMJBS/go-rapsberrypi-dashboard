# Go テンプレート: コーディング規約（Coding Standards）

このドキュメントは、チームでこのテンプレートを使う際の **合意事項（規約）** をまとめたものです。
「人の好み」になりやすい論点は増やさず、**Go の標準慣習 + 自動化で担保できる範囲** を中心にしています。

## 基本方針

- **迷ったら標準に寄せる**（Go のエコシステムで一般的なやり方を優先）
- **規約は“短く・守れる”** を優先（例外運用を前提にしない）
- **自動で直せるものは自動で直す**（フォーマット等）

## 自動で強制されるもの（ローカル/CI）

次はローカルの pre-commit と CI の両方で実行され、差分があれば落ちます。

- フォーマット: `gofmt`（+ import 整理: `goimports`）
- 静的解析: `golangci-lint`（設定は [/.golangci.yml](../.golangci.yml)）
- 既知脆弱性チェック: `govulncheck`

日常の確認は `mise run fmt` / `mise run lint` / `mise run vuln` を使います。

## 命名

- パッケージ名は **短く・小文字**・意味が明確（`util` の乱用は避ける）
- エクスポート（公開）識別子は、利用側から意味が通る名前にする
- 省略は一般的なもののみ（`ctx`, `req`, `resp`, `id` など）。チーム内ローカル略語は避ける

## エラー処理

- エラーは **握りつぶさない**（境界で返す/ラップする/ログする）
- ラップは `%w` を使う

```go
return fmt.Errorf("load config: %w", err)
```

- ログは「入口/境界（main、HTTP handler、job runner など）」でまとめて行い、ライブラリ層で無闇に `log.Printf` しない

## `context.Context`

- `context.Context` は **第1引数**
- `context.Background()` / `context.TODO()` を使うのはエントリポイント（`main`、テスト等）に限定
- 値の詰め込み（`context.WithValue`）は最小限にし、まずは明示的な引数を優先

## コメント / ドキュメント

- エクスポート（公開）する識別子には **godoc 形式**のコメントを書く（識別子名で始める）

```go
// Server starts the HTTP server.
type Server struct {}
```

- 「なぜそうするか」を書く（コードから読み取れることの重複説明は避ける）

## パッケージ構成（成長に合わせた目安）

小さい間は `main.go` でもOKですが、育ってきたら以下を推奨します。

- 実行ファイル: `cmd/<app>/main.go`
- アプリ内部専用: `internal/...`
- 共有ライブラリ（本当に必要な場合のみ）: `pkg/...`

## テスト

- テーブル駆動テストを基本にする
- 失敗時に原因が分かるように、`t.Run` 名や出力を丁寧にする
- 外部依存（ネットワーク/時刻/乱数）は注入可能にし、ユニットテストを安定させる

## 依存関係

- 追加依存は慎重に（まず標準ライブラリでできないか検討）
- 依存を追加する場合は「なぜ必要か」を PR の説明に書く
