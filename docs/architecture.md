# アーキテクチャ方針（Raspberry Pi 1B / FrameBuffer / rclone）

このドキュメントは、Raspberry Pi 1B（非 X11 環境）で動作する軽量ダッシュボードを実現するための **アーキテクチャ方針** をまとめたものです。

## 目的

- Raspberry Pi 1B 上で、軽量に常時表示できるダッシュボードを動かす
- 非 X11 環境で、Linux FrameBuffer（例: `/dev/fb0`）に直接描画する
- 以下を表示する
  - 時計（秒単位で更新）
  - 現在地（固定の緯度経度）の天気
  - Google Drive の特定フォルダにある画像（複数枚からランダム表示、定期更新）

## 非目的（やらないこと）

- X11/Wayland などの GUI スタックに依存すること
- Google Drive API をアプリ内に実装して認証/同期まで行うこと（同期は rclone に外出し）
- 高度な UI 機能（複雑なアニメーション、ページ遷移、入力 UI、設定画面など）

## 前提・制約

- 実行環境: Raspberry Pi 1B（リソースが限られる）
- 画面: SXGA 1280x1024、24bpp（ただし実際のピクセル配置は機種/設定で異なる可能性がある）
- 画像描画: `/dev/fb0` に対して mmap で書き込み
- ネットワークは不安定になり得る（天気取得失敗に耐える）
- buildrootでビルドしたSysVinitベースの環境で動かすことを想定
- overlayFSでreadonly化された環境で動かすので、変更はtmpfsに逃げる。
よって、OOMを回避するため、ファイルの書き込みは必要最小限にする（例: 天気キャッシュ、画像の縮小済みキャッシュなど）

## 主要な設計判断

### 1) 描画は「1プロセス・単純なループ」

- 描画は 1秒周期で「1フレーム生成 → FrameBuffer へ転送」を繰り返す
- 画面合成はオフスクリーン（`image.RGBA`）に行い、最後に FB のピクセル形式へ変換して一括転送
- これにより、UI ツールキット不要・依存最小・動作が読みやすい

### 2) 更新周期は責務ごとに分離し、キャッシュ前提で動かす

- 時計: 1秒更新
- 天気: 10〜30分更新（API 呼び出し頻度を抑える）
- 画像: 30秒〜数分で切替、数分でディレクトリ再スキャン
- 外部取得（天気）は「失敗しても最後のキャッシュを表示」して画面が壊れないようにする

### 3) Drive は rclone に外出しし、表示アプリはローカルだけを見る

- Google Drive の認証/差分同期は rclone に任せる
- 表示アプリはローカルディレクトリ（同期先）をスキャンして画像を選ぶだけにする

この分離により、
- アプリ側の依存と複雑性を最小化
- OAuth トークン運用などの泥臭さを表示プロセスから排除
- 同期が失敗しても表示側が落ちない

## コンポーネント構成

実装の大枠は以下です。

- `cmd/dashboard`: エントリポイント（フラグ・ディレクトリ作成・シグナル処理）
- `internal/app`: 更新スケジューリング、状態管理、レンダリングの統合
- `internal/fb`: FrameBuffer の open/ioctl/mmap と転送（24bpp/32bpp）
- `internal/photos`: ローカル画像の列挙、デコード、画面サイズへの整形
- `internal/weather`: 天気 API 取得、キャッシュの保存/読み込み
- `internal/widgets`: 時計（7セグ）や小さなテキスト描画

## データフロー

### 起動時

1. 設定（フラグ）を読み込み
2. `photos_dir` / `cache_dir` を作成
3. FrameBuffer を open + mmap
4. 天気キャッシュを読み込み（存在すれば表示に反映）
5. 写真ディレクトリをスキャンし、最初の画像を選ぶ
6. 以降、描画ループ開始

### 実行中（定常状態）

- 毎秒: 現在時刻で時計を更新し、フレームを再生成して転送
- 天気更新周期: API 取得 → 成功ならキャッシュ保存 → 表示更新
- 画像切替周期: ローカル候補からランダム選択 → デコード/整形 → 表示更新
- 画像再スキャン周期: 同期ディレクトリを再スキャンして候補一覧更新

## FrameBuffer に関する方針

- `FBIOGET_VSCREENINFO` / `FBIOGET_FSCREENINFO` を読み、`line_length` と色の bitfield（offset/length）を使ってピクセル配置を判断する
- 24bpp の場合でも「RGB24 とは限らない」ため、offset ベースで R/G/B の並びを決める
- まずは 24bpp/32bpp（byte-aligned の 8bit チャンネル）を対象とし、未知の形式は明示的にエラーにする

## パフォーマンス方針

- 1フレームは 1280x1024x3byte ≒ 3.9MB 程度なので、1Hz の全画面転送は現実的
- CPU を食いやすいのは画像デコード/リサイズなので、
  - 画像切替タイミング以外は重い処理をしない
  - 将来的には「縮小済みキャッシュ（画面サイズへ整形済み）」を保存して再利用できる余地を残す

## 障害時のふるまい（落ちにくさ）

- 天気取得に失敗しても、最後のキャッシュを表示し続ける
- 写真が 0枚でも、背景＋時計＋エラーメッセージは表示できる
- FrameBuffer を開けない場合は致命（起動失敗）。開発機での実行は想定外

## 運用（rclone）

- Drive → ローカル同期は（cron / BusyBox `crond` / systemd timer 等で）定期実行する
- 表示アプリは同期の成否を直接は気にせず、ローカルの状態だけを参照する

具体例は [docs/run-on-raspberrypi.md](run-on-raspberrypi.md) を参照してください。

## Buildroot での rclone について

Buildroot の提供パッケージはバージョン/設定によって変わるため、「使える前提」で固定せずに、次の手順で確認する。

### 提供されているかの確認

- Buildroot ツリーで `make menuconfig` を開き、パッケージ一覧に `rclone` が存在するか確認する
- あるいは、Buildroot ツリーで `BR2_PACKAGE_RCLONE` を検索し、`Config.in` / `package/` 配下に定義があるか確認する

提供されている場合は、基本的に Buildroot のパッケージとして組み込むのが再現性が高い。

### 提供されていない場合の代替案

優先度順に、軽量性・運用の簡単さを重視して選ぶ。

1. **同期を別マシンへ外出し**
  - LAN 内の別サーバ/PC で `rclone sync` を実行し、Pi 側へ `rsync/scp` 等で写真ディレクトリだけ配布する
  - Pi 側のダッシュボードは「ローカル写真ディレクトリを読む」だけのまま維持できる
2. **rclone を Buildroot に追加（カスタムパッケージ）**
  - Buildroot が rclone を持っていない場合でも、外部ツリー（BR2_EXTERNAL）でパッケージを定義して組み込める
  - Go のビルド環境/サイズに注意（必要ならターゲット向けに最小構成へ寄せる）
3. **rclone を事前ビルドした単体バイナリとして同梱**
  - CI などで `linux/armv6` 向けに rclone をビルドし、rootfs overlay へ配置して運用する
  - ただし Buildroot の再現性（ソース/バージョン固定）やセキュリティ更新の管理は別途必要
4. **Drive API 同期を自前実装（非推奨）**
  - OAuth トークンの初期取得/更新や API 差分取得が複雑になり、軽量化の方針に反するため、最後の選択肢とする
