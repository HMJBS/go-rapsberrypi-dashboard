# weather.Fetch のアイコン対応

## 背景

現状の `weather` パッケージでは、現在天気の取得とアイコン決定が別 API になっている。

- `Client.Fetch()` が現在の天気情報を取得する
- `Weather.GetIcon()` が日出・日没情報を追加取得し、表示用アイコンを決定する

この構成により、呼び出し側で以下の手順を毎回組み立てる必要がある。

1. `Fetch()` で `Weather` を取得する
2. 取得した `Weather` に対して `GetIcon()` を呼ぶ
3. それぞれのエラーを個別に扱う

特に `internal/app/app.go` では、初期キャッシュ読込時と定期更新時の両方でこの分離を意識する必要があり、責務が `weather` パッケージから呼び出し側へ漏れている。

また、昼夜判定に必要な日出・日没時刻が `Weather` に含まれていないため、キャッシュから復元した `Weather` だけではアイコン決定に必要な情報が完結していない。

## 目的

表示用途では、1 回の呼び出しで「現在天気」と「対応アイコン」をまとめて取得できるようにする。

これにより、呼び出し側は `weather` パッケージの内部手順を知らずに済み、`app.go` 側の実装とエラーハンドリングを単純化できる状態を目指す。

あわせて、アイコン決定に必要な日出・日没時刻も取得結果に含め、キャッシュ復元時にも追加 API 呼び出しなしで表示判断できるようにする。

## 方針

`weather` パッケージ側で、Open-Meteo の 1 回のリクエストに現在天気と daily 情報を同梱して取得する。

想定する責務は以下のとおり。

- 現在天気の取得
- 日出・日没の取得
- 天気コードと昼夜判定に基づくアイコン選択
- 呼び出し側で扱いやすい形での結果返却

Open-Meteo は `current` と `daily` を同一レスポンスで取得できるため、現在天気取得と日出・日没取得を別 API に分けない。

呼び出し側は、この取得結果をそのまま state に反映するだけにする。

## 変更案

### 1. `Fetch()` を 1 回の API 呼び出しで完結させる

`Client.Fetch()` のリクエスト内容を見直し、`current` に加えて `daily=sunrise,sunset` も同時に取得する。

これにより、現在天気取得後に別 API で日出・日没を問い合わせる必要をなくす。

### 2. `Weather` に日出・日没を保持する

キャッシュ復元時にも昼夜判定を再利用できるよう、`Weather` に以下の情報を追加する。

- `Sunrise time.Time`
- `Sunset time.Time`

これにより、`weather.json` に保存される 1 件のデータだけで、温度、天気コード、観測時刻、取得時刻、日出、日没までが完結する。

### 3. `GetIcon()` は廃止する

`Weather.GetIcon()` は、メソッド名から期待される責務と実際の役割がずれているため廃止する。

理由:

- `Weather` 自体はデータ保持に集中させたい
- 表示向けのアイコン解決は `weather` パッケージ側の組み立て責務として扱いたい
- 呼び出し側に `Weather` 取得後の追加手順を残したくない

そのため、呼び出し側が `Weather` を受け取ってから追加でアイコン取得処理を呼ぶ設計は採らない。

### 4. 表示向けの集約戻り値を追加する

表示用途では、`Fetch()` とは別に表示向けの集約結果を返す API を追加する。

例:

- `type Snapshot struct { Weather Weather; Icon image.Image }`
- `FetchSnapshot(ctx, lat, lon, tz)`

想定する役割分担は以下のとおり。

- `Fetch()` はキャッシュ対象になる `Weather` を返す
- `FetchSnapshot()` は表示に必要な `Weather` と `Icon` をまとめて返す
- キャッシュ復元時は保存済みの `Weather` から `Snapshot` を組み立てる補助関数を用意する

これにより、データ取得と表示組み立ての責務を `weather` パッケージ内に閉じたまま、`Weather` 自体はシリアライズしやすい構造として維持できる。

## app 側の変更方針

### 初期キャッシュ読込

キャッシュには `Weather` を保存し、その中に日出・日没も含める。

これにより、キャッシュ復元時は追加 API 呼び出しなしで保存済み `Weather` から表示用 `Snapshot` を組み立てられる。

### 定期更新

`updateWeather()` では、現在のように `Fetch()` と `GetIcon()` を分けて呼ばない。

代わりに、表示向けの `FetchSnapshot()` を 1 回だけ呼び、その結果を state に反映する。

これにより、段階的エラー処理は不要になり、更新処理は「表示に必要な天気情報を取得する」という 1 操作にまとまる。

## キャッシュの扱い

`weather.json` の保存内容は引き続き `Weather` のみとする。ただし、`Weather` 自体を日出・日没を含む形へ拡張する。

理由:

- `image.Image` はキャッシュ対象として重く、再利用価値も低い
- 日出・日没は昼夜判定に必要で、キャッシュからの復元時にも使い回せる
- Open-Meteo の 1 回のレスポンスで取得できるため、取得コストに対して保存価値がある
- キャッシュ対象を `Weather` に集約したまま必要情報を完結できる

そのため、キャッシュ復元時は保存済みの `Weather` から表示用 `Snapshot` を組み立てる。

## テスト方針

`internal/weather` にテストを追加し、少なくとも次を確認する。

1. 1 回の API 応答から、現在天気と日出・日没を含む `Weather` を取得できる
2. `FetchSnapshot()` が `Weather` と対応アイコンをまとめて返せる
3. `weather_code=0` で昼間なら晴れ昼アイコン、夜なら晴れ夜アイコンになる
4. 日出・日没を含む `Weather` をキャッシュ保存し、復元後も追加 API なしで同じ `Snapshot` を組み立てられる
5. タイムゾーン解釈と時刻パースが既存仕様を壊していない

HTTP 通信は `Client.HTTPClient` を差し替えてモックする。

## 期待される効果

- `app.go` 側の呼び出し手順が簡素化される
- 天気取得ロジックの責務が `weather` パッケージ内に閉じる
- エラーハンドリングが一元化される
- 初期キャッシュ読込と定期更新で同じ表示組み立て方針を使える
- キャッシュ復元時にも日出・日没のための追加 API 呼び出しが不要になる
- `GetIcon()` 廃止により API の責務が明確になる

## 実装時の注意点

- `Weather` 構造体の拡張に伴い、既存キャッシュとの後方互換をどう扱うか確認する
- 日出・日没は日単位で変わるため、キャッシュの有効期限判定は従来どおり `FetchedAt` ベースでよいか確認する
- `GetIcon()` を廃止したあと、呼び出し側に類似の組み立て処理を再流出させない
- 変更後も `mise run fmt`、`mise run lint`、`mise run vuln`、`mise run test` を通す
