# Go テンプレート: レビュー観点（Review Checklist）

このチェックリストは、PR レビューで **毎回迷わず同じ品質** を担保するための観点集です。
`gofmt/goimports`・`golangci-lint`・`govulncheck`・CI は機械的に担保される前提で、
ここでは主に **人の判断が必要な点** を確認します。

## 1. 目的とスコープ

- 変更の目的が PR 説明から理解できる
- スコープが適切（無関係な変更・リファクタが混ざっていない）
- 後方互換/破壊的変更の有無が明記されている（必要なら移行手順も）

## 2. 正しさ（Correctness）

- 要件を満たしている（境界条件/エッジケース含む）
- エラーが握りつぶされていない（失敗時の挙動が明確）
- リトライ/タイムアウト/キャンセルが必要な箇所で考慮されている

## 3. 設計と責務

- 変更が適切なレイヤに入っている（入口/ドメイン/外部I/Oなど）
- 関数/型の責務が小さく、命名が意図を表している
- 将来の拡張が必要な箇所で、過剰設計せずに拡張余地を残している

## 4. `context.Context` と並行処理

- `context` は第1引数で渡され、キャンセルが伝播する
- goroutine のライフサイクルが管理されている（リークしない）
- 共有状態がある場合、データ競合（race）の可能性がない

## 5. エラー/ログ

- エラーは `%w` でラップされ、呼び出し元が原因を追える
- ログは「入口/境界」で行い、ライブラリ層に散らばっていない
- ログに機密情報（トークン/秘密鍵/個人情報など）が出ない

## 6. テスト

- 重要な分岐やバグが入りやすい箇所にテストがある
- テストは安定していて、外部依存に引きずられない（時間/乱数/ネットワーク）
- 失敗時に原因が分かる（`t.Run` 名、アサーション、ログ）

## 7. 互換性・運用

- 設定/環境変数/フラグの追加がある場合、README 等に反映されている
- 失敗時の復旧や観測（メトリクス/ログ/エラー）が最低限考慮されている
- Raspberry Pi 等のターゲット環境（OS/arch）で問題が起きにくい作りになっている

## 8. セキュリティ

- 依存追加がある場合、必要性が説明されている
- 入力（ファイル/HTTP/CLI 引数など）のバリデーションが妥当
- `govulncheck` が PR/CI で走る前提になっている（必要なら結果を確認）

## 9. ドキュメント/レビューしやすさ

- README / docs に必要な更新が含まれている
- 変更が大きい場合、コミットが分割されている（レビュー容易性）

## レビュアー向けメモ（運用例）

- 「必須」: 1〜6 を毎回確認
- 「必要に応じて」: 7〜9 を変更内容に応じて確認
